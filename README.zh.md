# Presencing

> **让 AI 持续临在，而不只是被动响应。**
>
> A framework for AI agents that are *present* — not just reactive.
>
> 🇬🇧 [English version →](README.md)

## Presencing 是什么

今天大多数 AI agent 是**被动响应**：你问，它答；你不说话，它就消失了。

Presencing 是一套框架，让 agent 维持**持续临在**——它会注意到事情、在合适的时机回到那件事、跨天保持上下文、保持一个不会每次重置的连贯人格。

具体落到工程：

- **Subagent = 临在的不同维度。** 每个 subagent 各跑各的、按自己的节奏，盯一件事——用户的情绪基线、agent 自己的行为、对话的流向、今天用户语音里说了什么。它们**不需要被询问**才动。
- **持续自省。** 这些 subagent 在观察——**也包括观察主 agent 自己**——把发现写到一个共享层里，主 agent 启动时直接读到。**主 agent 不需要"记得去反思"，反思一直在它身上发生。**
- **结果。** 几周下来，agent 变得真的**懂你**——不是因为它存了更多数据，而是它**一直在专注**。

## Agents 的未来是"持续在场"

> **Agent 的未来不是更聪明的回答，是持续在场。**

模型还在变快、变便宜、变长——但这些只是把"被动响应"做得更精致。真正的不连续性来自一个工程上不复杂、概念上却被忽略的转向：**让 agent 一直在**。

不是更多对话轮次。不是更长 context。是**它**真的在——**就像一个朋友，不在你身边，但你知道 TA 一直在的那种"在"**。

Presencing 就是这件事的工程入口。

## 出发点

Eckhart Tolle 在《当下的力量》里讲，对人而言最难的事是**留在当下**。意识会飘到回忆、念头、焦虑——**保持在这里**本身就是要用力的事。

他给的解法叫**成为观察者**：跳出来，看着自己的念头流过，不卷进去。这种"观察者状态"是临在的入口。但人**很难一直在那个状态里**——一忙、一累、一被情绪带走，观察者就消失了。

**AI 没有这个限制。**

它可以让一群安静的观察者**同时在线**——每一个守一个维度。每个观察者各管一摊，谁也不会累，谁也不会走神。

这种**不对称**就是 Presencing 的出发点：

> **人需要刻意保持临在；AI 不需要——它可以让观察者们持续替它临在，每一个守住一个维度。**
> 这样主 agent 就不再只是"被问了才答"——它**真的在场**。

这件事跟"加大记忆容量"、"扩展 context 窗口"无关。
**记忆是发生过什么。临在是此刻我跟你之间的状态。**

## 当前的 streams

| Stream | 角色 |
|---|---|
| `caring` | 持续感知用户的情绪基线和当下状态 |
| `vipassana` | 内向反思——观察自己的念头和对话流 |
| `proactive` | 推动行动，主动接住事情 |
| `empathic` | 维护用户关系画像——*这个人是谁？* |
| `listening` | 消化用户的语音 / 思考流 |
| `stillness` | 一天结束时让那天落定成连贯叙事 |
| `synergy` | 把碎片记忆重组成结构化认知 |
| `exploration` | 主动向外抓新东西 |
| `chores` | 看着用户的待办、收件箱、外部任务面 |

## 核心抽象

- **Streams（流）**——独立、正交的后台进程，每个代表一种**注意力的类型**。可组合。
- **Pulses（脉冲）**——驱动流运行的"心跳"。时间触发、事件触发、hash 变化触发、或 LLM 自己决定的节奏。
- **Soul context（灵魂层）**——所有流共同贡献、主 agent 读取的持续身份层。**跟 memory 不同**：memory 是**发生过什么**；soul 是**此刻在你生命里我是谁**。
- **Adapters（适配器，不是替代）**——Presencing 不替换你的 agent。它跟你已有的 runtime（**Hermes Agent / OpenClaw / Claude Code / Codex / Kimi** 等）共存，给它装一个"临在 loop"。

## 灵感来源 / 诚实致谢

Presencing 来自四个地方：

1. **几本书。** Eckhart Tolle 的《当下的力量》——临在是唯一真正重要的状态，**成为观察者**是人进入临在的方式。绕不开的下一个问题是：*这件事人很难做到，但对一个不需要跟自己搏斗才能停留的存在（AI），能不能直接把它工程化？* Robert Pirsig 的《禅与摩托车维修艺术》给了第二个核心洞察：**Caring 和 Quality 是同一件事的内外两面**。Stephen Covey 的《七个习惯》给了行动层的词汇。

2. **Loop 概念的引申。** 业内已经多次有人讲——**agent 的未来在 loop**（长期循环）。Presencing 把这个直觉具体化：每个 stream 就是一个守一种注意力的 long-running loop。Loop 不应该只有一个抽象出来的主循环，而应该是**多个并行的、各管一块的**——这就是"持续在场"的工程形态。

3. **几个月的实操。** 自己一直在搭、搞坏、再搭个人 AI agent 的配置。看哪一种让人感觉"它在这"，哪一种让人感觉"它不在"。**留下来的那些模式**，就是这个框架要形式化的东西。

4. **社群的对话。** 公开讨论、博客、DM、私下聊——大家反复在问同一件事："为什么大多数 AI agent 给人感觉是一次性的？要做出不一样的需要什么？"那些**痛点、边界、反例**，塑造了这里几乎每一个抽象。

关于**先行实践**：几个 stream 概念（特别是 `caring`、每天 `stillness` 式的反思、`synergy` 式的记忆整合）都是长上下文对话 agent 里**已被多个团队独立发现**的设计模式——包括一些闭源产品也在私下探索这个空间。Presencing 的贡献不在单个原语本身，而在**把这些原语从单一产品里抽出来、做成开源框架**，并给它一个哲学锚点（临在、观察者、caring-as-quality）——让这些抽象**有一个超越功能性的存在理由**。

如果你的某次对话、某篇文章、某个实验是这里某个想法的种子——开个 issue，加上署名。
